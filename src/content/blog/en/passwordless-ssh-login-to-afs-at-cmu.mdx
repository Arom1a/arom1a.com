Every CS student at CMU has to do their programming assignment on AFS (Andrew File System),
but typing your password every time when SSH-ing to the server is annoying,
not mentioning that if you have a complicated password like I do.
However, you are different: you are a 10x student and want to devote all your time to coding!

So you wonder, is it possible to setup ssh-key authentication like all other servers?
Unfortunatly, this is banned on AFS for some clever reasons [^reddit]. The gist is that ssh keys only give you access to the server, but not the file system, which requires Kerberos tickets [^cmu].

[^reddit]: https://www.reddit.com/r/cmu/comments/3jsd1c/anyone_know_how_to_set_up_passwordless_ssh/

[^cmu]: https://computing.cs.cmu.edu/help-support/kerberos

Some guides already exist, such as
[ssh to AFS without credentials — Fanpu Zeng](https://fanpu.io/blog/2018/ssh-to-AFS-without-credentials),
but that post is fairly outdated now, and some of the details are incomplete.
So this post documents a more reliable approach of setting up automatic SSH login at CMU.

> Kerberos is a centralized authentication protocal *(that AFS uses)* where users verify their identity once with a trusted authority to receive a temporary, encrypted digital ticket. This ticket then acts as a secure credential, granting the user seamless access to various network resources without ever repeatedly exposing their actual password.

Actually you don't need to understand what Kerberos is. Just follow the tutorial below!

## Preparation

First, the Kerberos client needs to be installed on your machine.

```sh
# ! package names vary for other package managers, check before you proceed

# Homebrew
brew install krb5
# Arch
pacman -S krb5
```

## Configuration

You also need a Kerberos configuration to let the client understand the CMU realm.
Check if there already exist one at `/etc/krb5.conf` before we move on. If there is, remember to back it up.
Then, we just grab the one used by AFS by

```sh
scp <andrewID>@unix.andrew.cmu.edu:/etc/krb5.conf /etc/krb5.conf
```

After this, add a host entry in your SSH config at `~/.ssh/config` (remember to replace the Andrew ID!):

```txt
Host cmu
  HostName unix.andrew.cmu.edu
  Port 22
  User <andrewID>
  GSSAPIAuthentication yes
  GSSAPIDelegateCredentials yes
```

Now you have set up the basic Kerberos authentication! Test the setup with following commands:

```sh
# list tickets before auth
klist
# obtain a ticket; this should ask for your password
kinit <andrewID>
# list tickets again
klist
# log in AFS with the ticket
ssh cmu
```

Some other commands you might find useful [^useful]:

[^useful]: https://club.cc.cmu.edu/talks/spring14/kerb_and_afs.txt

```sh
# renew renewable tickets, but will be useless at the end of the next section
kinit -R

# if you happen to use the vscode remote ssh plugin
# this opens the remote ssh folder directly
code --folder-uri "vscode-remote://ssh-remote+cmu/afs/andrew.cmu.edu/<folder-path>"
```

## No More Passwords!

With the basic setup, I know some of you are still not satisfied.
You would need to input your password at least once per day
either because the ticket disappears after system reboots or expires after 24 hours.
Even if you choose to execute the `-R`, it is very awkward.
However, we could do better!

For mac users, this can be achieved fairly smoothly; on linux however, it is a bit nontrivial. But c'mon, you are already using linux!

### Mac

Just use `kinit <andrewID> --keychain` !!!

This is because the Kerberos client comes with keychain integration (at least on my machine...). Read more in the man page.

### Linux

Unfortunately, the `--keychain` option is not available for Linux users. Instead, we would use `gnome-keyring` and `secret-tool` to store and retrieve our credentials so that we don't need to type them out when getting a ticket.
We could just pipe the password into `kinit` afterwards.

The following steps assume you already have `gnome-keyring` and `secret-tool` installed.

```sh
# store the password in the keyring
# this will asks you to input the password
secret-tool store --label "Andrew Account" user "<andrewID>"

# later when asking for a ticket, pipe the password to kinit
secret-tool lookup user "<andrewID>" | kinit <andrewID>
```

## Bonus: sshfs

The vscode remote ssh extension suffers from downloading the vscode server binary upon every update and blows up your disk quota very quickly, and SFTP syncing is such a pain that I would rather use `rclone`.
So, in this bonus section, we are going to explore `sshfs`.

To start using it, you need to install `sshfs` or `fuse-sshfs` first using your package manager. ~~On macOS especially, it requires you to enter recovery mode to enable the file system. Unfortunately, my touch id button is broken and I could not enter that mode, so I have never experienced sshfs on my mac.~~ (As of v5.0 I think, macFUSE changed backend [^backend] so mac 15.4 or later versions do not need to enter recovery mode post-installation anymore. Just `goto` [macFUSE home](https://macfuse.github.io/) and grab both `macFUSE` and `SSHFS` packages. After that, pass `-o backend=fskit` to `sshfs` to bypass the recovery mode.)

[^backend]: https://github.com/macfuse/macfuse/wiki/FUSE-Backends

But luckily I also useArchBtw&trade;.

To use sshfs, first create an empty folder as a mount point (we will name it `sshfs-mnt`). Then run something like

```sh
mkdir sshfs-mnt # to create the directory just in case
sshfs -o idmap=user cmu:private/15150/hofs sshfs-mnt
```

to mount an AFS directory locally. Then you would be happy to navigate the directory as if it is a local folder!

The `-o idmap=user` option maps remote file UID/GIDs values to your local user so that git can function normally in the mounted directory.

Later, when you want to unmount the directory, you could do

```sh
umount sshfs-mnt
```

However, sometimes you may find yourself unable to unmount it because the directory is busy. Then you could use

```sh
fuser -vm sshfs-mnt
```

to check which programs are using it, and then possibly use `kill` to kill the programs.
